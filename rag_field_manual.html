The big picture

Two pipelines wearing one name

The single most useful thing to understand before any detail: RAG is not one flow, it is two. They run at different times and most confusion comes from blurring them.

Pipeline A — Indexing (offline, runs ahead of time)

You take your documents and prepare them for fast search: load and extract the raw text, split it into chunks, turn each chunk into a vector with an embedding model, and store those vectors in an index. This happens once (and again whenever documents change). It touches no user and no question.

Pipeline B — Querying (online, runs per question)

A user asks something. You embed their question with the same model, search the index for the nearest chunks, optionally rerank them, paste the best ones into a prompt as context, and let the language model write a grounded answer. This is the part the user feels, and it must be fast.

The mental model that makes it stick Indexing is writing the index at the back of a textbook. Querying is using that index to flip straight to the right pages, then writing an answer in your own words from only those pages. The embedding model is the thing that decides which entries are "about the same topic" — and you must use the identical model on both sides, or the index and the question speak different languages.

The playground below carries one real question — "Does Humira need prior authorization?" — through the whole journey. Step through it and watch the shape of the data change at every hop: text becomes vectors, vectors become ranked chunks, chunks become a prompt, the prompt becomes an answer.

Stage 1 · indexing

Ingestion & extraction

Before anything intelligent happens, you need clean text. Real documents are PDFs, Word files, HTML, scans, slides — messy containers. Extraction pulls readable text (and ideally structure: headings, tables, lists) out of them.

raw file → extract → plain text / markdown

Quality here sets the ceiling for everything downstream. Garbled tables, missing headers, or columns merged into nonsense will silently poison retrieval no matter how good your later stages are. Structure-aware extraction (keeping a heading attached to its paragraph) pays off later because it gives chunking natural seams to cut along.

C# — extract before you chunk

// In the running example, a service wraps the extractor.
// It turns a PDF into markdown text the rest of the pipeline can read.
var extractor = new DocumentExtractor(config);
string text = await extractor.ExtractAsync("Policies.pdf");
// text now holds the document as clean, chunk-ready markdown

Tools & open-source alternatives

  • Used here: Azure Content Understanding (managed PDF/scan → markdown, handles tables and layout).
  • Open source: DoclingUnstructuredApache TikaPyMuPDFpdfplumber — Docling and Unstructured are the strongest for layout-aware extraction; PyMuPDF is fast and simple for clean text.
Common trapScanned PDFs are images, not text. They need OCR (e.g. Tesseract, or a vision model) before extraction, or you index empty strings and wonder why retrieval finds nothing.
Stage 2 · indexing

Chunking

Embedding models and LLMs both have limits, and a whole document is too coarse to retrieve precisely. So you split text into chunks — the unit that gets embedded, stored, retrieved, and handed to the model. Chunking quietly decides retrieval quality more than almost any other knob.

one long text → split → many chunks (each a few hundred words)

The tension is simple. Chunks too small lose context — a sentence about "the 5-day window" with no idea what window. Too large and a single chunk covers several topics, so its vector becomes a muddy average and you waste the model's context on irrelevant text. You are tuning for "one chunk = one coherent idea."

The main strategies

StrategyHow it cutsGood forCost
Fixed-sizeEvery N tokens/words, blind to meaningUniform text, FAQs, a fast baselineLowest
Fixed + overlapFixed, but repeat the last 10–20% into the next chunkTechnical/legal text where ideas cross boundariesLow
RecursiveSplit on paragraphs, then sentences, then words, until it fitsMost prose — the usual go-to defaultLow
SemanticEmbed sentences, cut where meaning shiftsLong narratives, research papersHigh (embeds while chunking)
Structure-awareRespect headings/sections/tablesManuals, structured docsMedium
What the research actually saysRecursive token chunking is the reliable default and often beats fixed-size on precision. But "fancier = better" is not guaranteed: several 2025 studies found a plain fixed size of ~200 words competitive with semantic chunking on real datasets, and semantic chunking sometimes underperformed by introducing noise. Translation: pick recursive with overlap as a baseline, then let your eval harness (Stage 9) tell you the truth for your data. A common starting point: ~512 tokens with 50–100 tokens of overlap, or ~200–500 words with 10–20% overlap.

C# — a transparent word-based chunker with overlap

public static IEnumerable<string> Chunk(string text, int size = 200, int overlap = 40)
{
    var words = text.Split(' ', StringSplitOptions.RemoveEmptyEntries);
    int step = Math.Max(1, size - overlap);
    for (int i = 0; i < words.Length; i += step)
    {
        yield return string.Join(' ', words.Skip(i).Take(size));
        if (i + size >= words.Length) break;
    }
}

Stage 3 · indexing & querying

Embeddings

An embedding model turns a piece of text into a list of numbers — a vector — positioned in a high-dimensional space so that texts with similar meaning land near each other. This is the trick that lets a computer search by meaning instead of by exact words.

"prior authorization needed?" → embed → [0.021, -0.118, 0.067, … ] (e.g. 1536 numbers)

So "PA required before dispensing" and "needs prior authorization" end up close together even though they share almost no words, while "delivery takes 2 days" sits far away. Closeness is measured by a similarity metric — usually cosine similarity (the angle between two vectors; 1.0 = same direction, 0 = unrelated). Dot product and Euclidean distance are the other common choices.

The rule you must not breakEmbed the documents and the user's question with the same model. Vectors from two different models live in incompatible spaces — comparing them is meaningless, and retrieval silently returns garbage. If you change the embedding model, you must re-index everything.

The demo embeds a query and four chunks into a tiny 5-concept space (real models use hundreds or thousands of dimensions, but the math is identical) and ranks them by cosine similarity. Change the query's emphasis and watch the ranking reshuffle.

Stage 4 · indexing

The index & the vector database

You now have a vector per chunk. Storing them is easy; searching millions of them in milliseconds is the hard part. That's what a vector index — and the database around it — is built for.

chunks + vectors + metadata → upload → searchable index

Why a special index at all?

The honest way to find the nearest vectors is brute force: compare the query to every stored vector. That's exact but slow — fine for thousands of chunks, hopeless for millions. So production systems use Approximate Nearest Neighbor (ANN) indexes that trade a sliver of accuracy for enormous speed.

  • Flat / brute force — exact, simple, slow. Great for small corpora.
  • HNSW (Hierarchical Navigable Small World) — a layered graph you "navigate" toward the closest neighbors. The dominant default: fast, high recall, memory-hungry.
  • IVF / PQ — partition the space into buckets and/or compress vectors. Scales to billions with less memory, slightly lower recall.
Metadata earns its keep hereStore more than the vector: a stable chunk id, the source document, page, and any access tags. Stable ids make retrieval debuggable and evaluation possible (Stage 9). Access tags enable permission-aware retrieval — a user only ever retrieves what they're allowed to see, which is the difference between a toy and something you can put in a regulated workplace.

C# — upload chunks with their vectors (Azure AI Search)

await searchClient.UploadDocumentsAsync(new[]
{
    new SearchDocument
    {
        ["id"]            = "policy-p3-c2",   // stable, debuggable
        ["content"]       = chunkText,
        ["contentVector"] = vector,            // the embedding
        ["source"]        = "Policies.pdf"
    }
});

Stage 5 · querying

Retrieval

The user has asked something. You embed their question with the same model, then ask the index for the top-k nearest chunks. This is the heart of the "R" in RAG, and where most quality problems live.

question → embed → query vector → ANN search → top-k chunks + scores

C# — a vector query for the 3 nearest chunks

var qVector = (await embeddingClient
    .GenerateEmbeddingAsync(userQuery)).Value.ToFloats().ToArray();

var options = new SearchOptions
{
    VectorSearch = { Queries = { new VectorizedQuery(qVector)
    {
        KNearestNeighborsCount = 3,        // top-k
        Fields = { "contentVector" }
    }}}
};
var hits = await searchClient.SearchAsync<SearchDocument>(null, options);

The three levers that decide retrieval quality

  • top-k — how many chunks to pull. A recall-vs-(precision + cost) dial: bigger k catches more but adds noise and tokens on every query. 3–5 is a common default; prove the right value with evaluation.
  • Hybrid search — combine dense vector search (meaning) with sparse keyword search (exact terms like a drug name or code, where BM25 shines). Fusing both beats either alone for most real corpora.
  • Reranking — retrieve a wider net (say 20), then a cross-encoder reranker reads each (question, chunk) pair together and rescores them, promoting the truly relevant to the top. Slower per item, far more accurate than the first-pass vector score, so you only run it on the shortlist.
Bi-encoder vs cross-encoder, in one breathThe embedding model is a bi-encoder: it encodes query and chunk separately, so it's fast and cacheable but coarse. A cross-encoder reranker reads both together with full attention — much sharper relevance judgement, much slower. The standard pattern: bi-encoder to retrieve cheaply and widely, cross-encoder to rerank the shortlist precisely.

Other retrieval boosters you'll meet: query rewriting / HyDE (rephrase or hallucinate an ideal answer, then search with that), and metadata filtering (restrict the search to a document set or a user's permissions before scoring).

  • Hybrid engines: OpenSearch / Elasticsearch kNNWeaviateVespa.
Stage 6 · querying

Augmentation — building the prompt

Retrieval gave you the best chunks. Augmentation is the unglamorous-but-decisive step of assembling them into a prompt that forces the model to answer from the context, not from its own memory.

top chunks + question → template → one augmented prompt

A good RAG prompt does three jobs: it presents the retrieved context clearly (often numbered, so the model can cite [1], [2]), it instructs the model to use only that context and to admit when the answer isn't there, and it then poses the user's question. That "say you don't know" instruction is your first line of defense against hallucination.

C# — assemble grounded context with citation markers

string context = string.Join("\n\n",
    topChunks.Select((c, i) => $"[{i + 1}] {c.Content}"));

string prompt = $"""
    Answer using ONLY the context below. Cite sources as [n].
    If the answer is not in the context, say you don't know.

    Context:
    {context}

    Question: {userQuery}
    """;
"Lost in the middle"Models attend most to the start and end of a long prompt and can overlook material buried in the middle. So order matters: put the strongest chunk first, keep the context tight, and don't drown the signal by stuffing in a huge k. This is why precision (Stage 9) matters, not just recall.
Stage 7 · querying

Generation

Finally, the language model reads the augmented prompt and writes the answer — grounded in the retrieved context, ideally with citations the user can verify.

augmented prompt → LLM → grounded answer + citations

C# — call the chat model

var chat = client.GetChatClient("gpt-4.1-mini");
var result = await chat.CompleteChatAsync(
    new SystemChatMessage(systemPrompt),
    new UserChatMessage(prompt));
string answer = result.Value.Content[0].Text;

Generation choices that matter: a low temperature (near 0) keeps answers deterministic and faithful, which is what you want for factual grounding; the system prompt sets the persona and hard rules; and a smaller, cheaper model often suffices once retrieval is doing the heavy lifting — the whole point of RAG is that the model doesn't need to know the answer, only to read it well.

RAG's superpower, restatedBecause the answer comes from retrieved text, you get current, domain-specific, citable responses without retraining the model — and you can point at exactly which chunk each claim came from. That traceability is why regulated industries reach for RAG.

Stage 8 · querying

The agent layer

Plain RAG retrieves once and answers once. An agentic layer lets the system decide when and what to retrieve, call tools, and chain steps — turning a single lookup into a small workflow.

In the running pharmacy example, four agents run in sequence, each adding a section and passing its work to the next: an intake agent extracts the structured details, a clinical agent checks safety (and searches the knowledge base for formulary info), an operations agent finds prior-auth and delivery rules (another knowledge-base search), and an orchestrator compiles the final report. Retrieval here isn't a single front-door step — it's a tool two of the agents call when they need grounding.

request → intake → clinical → operations → orchestrator → report

This is the leading edge of RAG: agentic retrieval, where an LLM plans queries, decides how many searches to run, hits multiple sources, and assembles structured results — instead of one fixed query at the start. More power, more moving parts, and more to evaluate.

Stage 9 · the part that separates toys from systems

Evaluation

Everything above can run end-to-end and still be quietly wrong. Evaluation is how you prove the system works — and, more importantly, how you catch the moment a change breaks it. This is the stage that turns "I tinkered with RAG" into "I built RAG with measurable quality and regression gates."

Evaluation splits cleanly in two, because RAG has two things that can fail independently: the retriever (did it fetch the right chunks?) and the generator (did it answer faithfully from them?). You measure them separately so you know which one to fix.

Retrieval metrics — graded against a golden set

A "golden set" is a hand-built list of questions, each tagged with which chunk(s) should come back. For each query you run the real search, mark each returned chunk relevant or not, and compute:

  • Hit@k / Recall@k — did any (or all) of the right chunks make the top k? Did we find it at all?
  • MRR — how high did the first relevant chunk rank? Rewards putting the good stuff at position 1.
  • Precision@k — of the k chunks returned, what fraction were relevant? Measures noise.
  • NDCG — rank-quality with graded relevance, discounting hits that appear lower down.

Generation metrics — usually scored by an LLM judge

  • Faithfulness / groundedness — is every claim supported by the retrieved context, or did the model invent things? (The inverse of hallucination.)
  • Answer relevancy — does the answer actually address the question?
  • Context precision / recall — was the retrieved context on-topic, and did it contain everything needed?
The move that signals seniorityWire thresholds into your test suite so the build fails when a metric drops — a regression gate. Now "should I change chunk size / top-k / the embedding model?" stops being an argument and becomes an experiment: change it, re-run, read the report. A green "evals" badge with a real number ("Hit@3 = 0.91 across 34 graded queries") is worth more than any adjective.

C# — a retrieval metric as a pure, testable function

public static double ReciprocalRank(IReadOnlyList<bool> relevance)
{
    for (int i = 0; i < relevance.Count; i++)
        if (relevance[i]) return 1.0 / (i + 1);  // rank of first hit
    return 0.0;                                // nothing relevant found
}

// As an xUnit regression gate:
[Fact]
public void HitAtThree_AboveThreshold()
    => Assert.True(suite.HitAtK(3) >= 0.70,
        $"Hit@3 dropped to {suite.HitAtK(3):P0}");

Never think about it again

The one-screen cheat sheet

If you remember nothing else, remember this.

  • RAG = two pipelines. Index documents offline; answer questions online. Same embedding model on both sides, always.
  • Chunk so one chunk = one idea. Recursive + overlap is the safe default; let evaluation pick the size.
  • Embeddings turn meaning into position. Cosine similarity = "how close in meaning."
  • The index (HNSW & friends) makes searching millions of vectors fast by being approximately, not exactly, right.
  • Retrieval = embed the question, pull top-k. Add hybrid search for exact terms and a cross-encoder reranker for precision.
  • Augment by pasting the best chunks into a prompt that says "use only this, cite it, admit when you don't know."
  • Generate at low temperature; the model reads the answer rather than recalling it.
  • Agents turn one retrieval into a planned workflow with retrieval-as-a-tool.
  • Evaluate retriever and generator separately, against a golden set, with thresholds that fail the build. This is the difference between a demo and a system.
Prove you own itFor any stage, ask yourself: can I explain it, change it, and debug it without help? The fastest way to find the gap is to break something on purpose — change top-k or a threshold, predict what each metric will do, then run it. When your prediction matches reality, it's yours.
